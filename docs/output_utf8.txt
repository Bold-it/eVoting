Backend Build Phases
HTU Secure Online Voting System
A sequenced, checklist-driven build order for all 8 backend modules
Team BlackNodes
CodeFest Hackathon  |  June 2026
How to Use This Document
This document sequences the 8 backend modules from the Backend Requirements Specification into 6 build phases, in dependency order ΓÇö each phase assumes the prior ones are working before it starts. Use the checklists in each phase as a literal sprint checklist: check items off as your team completes them, and do not start a phase's checklist until the previous phase's checklist is fully checked.
This document does not replace the Backend Requirements Specification or the SRS ΓÇö it tells you WHEN to build each piece those documents already specify in full detail. Refer back to them for exact schemas, endpoint contracts, and business rules.
PHASE 1
Foundation ΓÇö Authentication & Access Control
Module(s): AuthModule
Nothing else in the system can be meaningfully tested without this module working first. Build and test it in complete isolation before touching anything else.
1.1 What This Phase Delivers
Voter login: Student ID + token ΓåÆ short-lived, election-scoped JWT
Admin login: email + password ΓåÆ access token + refresh token
JWT verification guard, used by every other module from Phase 2 onward
Role-based guard distinguishing super_admin, election_officer, and the voter scope
Refresh token rotation and admin logout (revocation)
1.2 Checklist
AdminUser and RefreshToken tables migrated and seedable with a test admin account
POST /api/v1/auth/admin/login returns access + refresh tokens for valid credentials
POST /api/v1/auth/admin/login returns identical error/timing for wrong password vs. non-existent email (no user enumeration)
POST /api/v1/auth/voter/login rejects invalid token, used token, and wrong-election token distinctly per the API contract
JWT Guard rejects expired and tampered tokens with 401
Role Guard rejects a voter JWT calling any admin endpoint, and vice versa
Rate limiting active on both login endpoints (5/15min) and returns 429 with Retry-After
Unit tests for: valid login, invalid credentials, expired token, wrong role ΓÇö all passing
PHASE 2
Election Setup
Module(s): ElectionModule, CandidateModule, VoterModule
These three modules are grouped because they are all admin-side configuration ΓÇö nothing here involves a real vote yet. They can be built in parallel by different teammates once Phase 1 is solid, since they don't depend on each other, only on Auth.
2.1 What This Phase Delivers
Election creation, position management, counting-method configuration (plurality / simple majority + fallback)
Election open/close state machine with the full readiness checklist enforced server-side
Candidate CRUD with attached portfolio (manifesto, credentials, photo)
Voter roll bulk upload (CSV) with row-level validation and rejection reporting
Voter token generation (hashes only ΓÇö plaintext never stored)
Config lock: positions, candidates, and voter roll become immutable once the election opens
2.2 Checklist
POST /elections creates a draft election; PATCH is blocked (423) once status leaves draft
Position creation rejects simple_majority without a majorityFallback value (400)
POST /elections/:id/open runs the full readiness checklist and returns structured 422 with unmetConditions if anything is missing
Candidate and portfolio CRUD enforces the manifesto character limit and strips HTML/script tags from free text
Candidate photo upload validates file type by content-sniffing, not just file extension
CSV voter roll upload returns per-row validation results (validRows / rejectedRows with reasons)
Token generation stores only argon2id hashes; a query of the database confirms no plaintext token exists anywhere
All three modules correctly return 423 Locked on any edit attempt after the parent election is open
PHASE 3
Getting Tokens to Voters
Module(s): NotificationModule
This phase depends on Phase 2 (voters and tokens must exist to email). It is its own phase because email delivery is asynchronous and has its own failure modes worth testing in isolation before voting depends on it.
3.1 What This Phase Delivers
Nodemailer configured against Google Workspace SMTP (smtp.gmail.com, port 587, STARTTLS, App Password auth)
BullMQ job queue: one email job per voter, enqueued on token generation, processed asynchronously
Retry logic: up to 3 attempts with exponential backoff before marking a delivery permanently failed
Delivery status tracking (pending / sent / failed) exposed to the admin dashboard
Resend-token endpoint, deliberately exempt from the election-open lock
3.2 Checklist
POST /elections/:id/voters/generate-tokens returns 202 immediately with a batch ID, not blocking on actual sending
A real test email successfully arrives via the Workspace account using an App Password (not a personal password)
Simulated send failure (e.g. invalid email) results in 3 retries, then status = failed, visible via GET /voters/delivery-status
POST /voters/:voterId/resend-token works correctly even while the election status is open
Email content reviewed: no plaintext word 'password', token only usable together with Student ID, sent only to the verified roll email
PHASE 4
The Critical Path ΓÇö Vote Casting
Module(s): VotingModule
This is the highest-stakes module in the entire system. Do not rush it, and do not consider it done until the concurrency test in the checklist below passes ΓÇö not just the happy path.
4.1 What This Phase Delivers
Ballot retrieval (positions, candidates, portfolios) for an authenticated voter session
Vote submission with: row-level lock on the voter's token, one-vote enforcement inside that lock, ballot payload validation, AES-256-GCM encryption with a fresh IV per vote, hash-chain append, all in a single atomic transaction
Idempotency-Key support so a dropped network response can be safely retried without double-counting
Vote receipt issuance and immediate session invalidation
4.2 Checklist
GET /elections/:id/ballot returns positions + candidates + portfolios correctly scoped to the voter's election
POST /votes succeeds exactly once per token and returns a receipt code not derived from vote content
Malformed ballots (missing position, unknown candidateId, duplicate position entries) are rejected with 400, never silently accepted
CONCURRENCY TEST: N simultaneous requests using the SAME token ΓÇö exactly one succeeds (201), all others receive 409, and exactly one VoteRecord exists afterward
CONCURRENCY TEST: N simultaneous requests using N DIFFERENT valid tokens ΓÇö all succeed, and the resulting hash chain has exactly N correctly linked entries
IDEMPOTENCY TEST: same Idempotency-Key submitted twice returns the original receipt, not a second vote
Database inspection confirms VoteRecord has no voterId, sessionId, or IP column ΓÇö anonymity is structural, not just policy
No application log line contains both a voter identity and any vote-cast detail in the same entry
PHASE 5
After the Polls Close
Module(s): TallyModule
This phase needs real VoteRecord data to test meaningfully ΓÇö run it against the votes produced during Phase 4's tests, plus a few hand-crafted scenarios for tie and majority-shortfall cases.
5.1 What This Phase Delivers
Hash-chain integrity verification across an election's full vote sequence
Per-position tally computation respecting each position's countingMethod
Tie detection and majority-shortfall detection, with abstentions excluded from the majority percentage denominator
Runoff creation, scoped to tied/top-two candidates and the original voter roll
Results publication, gated on integrity verification passing and every position being resolved
5.2 Checklist
POST /elections/:id/verify-integrity correctly returns verified: true on an untouched chain
Manually corrupting one VoteRecord's stored hash causes verify-integrity to return verified: false with the correct brokenAt index
GET /elections/:id/tally-review correctly computes a clear-winner scenario, a tie scenario, and a no-majority-reached scenario (test with crafted data for the latter two)
Abstention counts are confirmed excluded from the majority percentage calculation by direct test, not just code review
POST /elections/:id/results/publish is blocked (422) while any position is unresolved, and succeeds once all are resolved
GET /elections/:id/results returns 404 (not 403) for any election not yet in results_published status
PHASE 6
Threaded Throughout ΓÇö Audit Logging
Module(s): AuditModule
Unlike Phases 1-5, this module is not a discrete sequential step ΓÇö build a minimal version early (during Phase 1) and revisit it as each subsequent phase is built, since every module above needs to call into it.
6.1 What This Phase Delivers
Append-only audit log: every admin/system action recorded, with no update or delete path exposed anywhere
Vote-related audit entries record only the event ("vote cast"), never which voter or any vote content
Paginated, filterable retrieval restricted to super_admin only
6.2 Integration Points (revisit as each phase completes)
After Phase
Audit Events to Confirm Are Logged
1 ΓÇö Auth
Admin login success/failure, admin logout
2 ΓÇö Election Setup
Election created, position added, candidate added/edited, voter roll uploaded, tokens generated, election opened/closed
3 ΓÇö Notification
Token resend triggered (who triggered it, not which voter's content)
4 ΓÇö Voting
"Vote cast" event only ΓÇö explicitly verify no voter identity appears in this entry
5 ΓÇö Tally
Integrity check run (pass/fail), runoff triggered, results published
6.3 Checklist
AuditLogEntry has no PATCH or DELETE endpoint anywhere in the API
GET /audit-log is callable only by super_admin (403 for election_officer)
Every event in the table above is confirmed present in the log after exercising that phase's flows

1. Introduction
1.1 Purpose
This document specifies the functional and non-functional requirements for the COMPSSA Secure Online Voting System, developed by Team BlackNodes for the CodeFest Hackathon. The system enables eligible Computer Science Student Association (COMPSSA) members to securely cast votes in association elections (e.g. President, Vice President, Secretary, Treasurer) through a digital platform, replacing manual paper-based or unsecured voting methods.
The system is designed as a foundation that can scale beyond COMPSSA to other student associations, departments, or university-wide elections without architectural changes ΓÇö only configuration (voter list, candidate list, election rules) changes per deployment.
1.2 Scope
The system covers the full election lifecycle: election creation and configuration by an administrator, secure voter authentication, one-person-one-vote ballot casting, real-time tallying, and transparent, tamper-evident results publication.
In scope:
Admin-managed election setup (positions, candidates, voter roll, election window)
Secure, token-based voter authentication (no public self-registration)
Single-use voting tokens to enforce one vote per eligible voter
Encrypted, anonymized vote storage (vote content is never linked to voter identity in storage)
Tamper-evident vote logging (hash-chained vote records)
Real-time turnout monitoring dashboard for admins
Automated, auditable result computation and publication
Out of scope (for hackathon/MVP phase):
Biometric verification (fingerprint/facial recognition)
Blockchain network deployment (a lightweight hash-chain simulates blockchain-style integrity instead)
Multi-language localization
SMS gateway integration (token delivery is via email/admin export for MVP)
1.3 Intended Audience
CodeFest hackathon judges evaluating architecture, security, UX, and production readiness
Team BlackNodes developers (implementation reference)
COMPSSA electoral committee (system operators / admins)
COMPSSA student voters (end users)

2. Overall Description
2.1 Product Perspective
The system is a standalone, self-contained web application (with a mobile-responsive interface) composed of three logical layers: a Voter Web App, an Admin Dashboard, and a backend API with supporting services (authentication, vote processing, tallying, audit logging). It does not depend on COMPSSA's existing systems for this MVP but is designed with clean API boundaries so it could later integrate with HTU's official student records system for automatic voter roll import.
2.2 Product Functions (Summary)
Election creation and configuration by administrators
Bulk voter roll upload and unique token generation
Secure voter login using Student ID + one-time token
Ballot presentation and single vote submission per position
Vote encryption, anonymization, and tamper-evident storage
Real-time turnout statistics for administrators
Automated tallying and results publication after election close
Full audit trail of system and admin actions (not vote content)

Each voter has access to a device with a modern browser during the election window
Hosting provider offers HTTPS by default or via reverse proxy (e.g. Let's Encrypt)
3. System Architecture
3.1 Architectural Style
The system follows a modular, service-oriented architecture within a single deployable backend (a 'modular monolith') ΓÇö giving hackathon-feasible delivery speed now, with clean internal boundaries that allow extraction into independent microservices later without rewrites. This directly addresses the scalability criterion: the architecture demonstrates a clear growth path rather than a one-off script.
3.1.1 Core Services
ID
Requirement
Description
Auth Service
Identity & access
Validates Student ID + token pairs, issues short-lived JWT session tokens, enforces RBAC
Election Service
Election lifecycle
Create/configure elections, manage candidates, positions, and election windows
Voting Service
Ballot casting
Validates voter eligibility, enforces one-vote rule, encrypts and stores vote, appends to hash chain
Tally Service
Result computation
Aggregates encrypted votes after election close, computes results, verifies hash chain integrity before publishing
Audit Service
Action logging
Immutable log of all admin and system actions (election created, token issued, vote cast event ΓÇö not content, results published)
Notification Service
Token delivery
Sends voting-token emails to each voter on roll upload, handles resend requests, and tracks delivery status, via an asynchronous job queue so bulk sends don't block election setup
3.2 Scalability Strategy
Stateless API layer behind a load balancer ΓÇö horizontal scaling by adding instances under voting-day traffic spikes
Redis used for caching live turnout counts and rate-limiting, reducing direct database load during peak voting periods
Database writes for vote casting are wrapped in atomic transactions with row-level locking on the voter's token record, preventing double-vote race conditions under concurrent load
Asynchronous job queue for non-critical tasks (e.g. notification emails, audit log writes) so the vote-casting path stays fast
Architecture designed to extend from COMPSSA (hundreds of voters) to university-wide elections (thousands of voters) by scaling the same components ΓÇö no redesign required
3.3 High-Level Data Flow
Admin creates an election and uploads the voter roll.
System generates a unique, single-use voting token per voter and makes it available for secure distribution (e.g. admin export, email).
Voter logs in with Student ID + token; Auth Service validates and issues a short-lived session.
Voter views the ballot and submits selections; Voting Service validates eligibility and that the token is unused.
Vote is encrypted, detached from voter identity, and appended to the hash chain; the token is immediately invalidated.
Voter receives a vote receipt (proof of participation, not vote content).
After the election window closes, Tally Service verifies the hash chain integrity, then computes and publishes results.

5. Vote Counting Rules
These rules govern how a winner is determined once voting closes. They must be configured per election before it opens, locked once voting starts, and recorded in the audit log so the counting method can never be disputed after the fact. Quorum (minimum turnout to validate an election) is explicitly out of scope for this phase and is deferred to a future iteration.
5.1 Counting Method
Admin selects one counting method per position at election creation time. The method is then immutable for that election.
ID
Requirement
Description
Plurality
Most votes wins
The candidate with the highest vote count wins outright, even if under 50%. Best for races with 3+ candidates where a runoff isn't practical in the hackathon timeframe.
Simple Majority
Over 50% required
A candidate must receive more than 50% of valid votes cast for that position. If no candidate crosses 50%, the system automatically triggers the Majority Shortfall procedure (Section 5.3).
For the MVP demo, Simple Majority is the default and recommended mode for COMPSSA executive positions (President, VP, Secretary, etc.), since it best reflects legitimate-mandate expectations for student leadership roles.
5.2 Tie-Breaking Procedure
A tie occurs when two or more candidates receive the exact same vote count and that count is the highest for the position. The system must never resolve a tie silently or arbitrarily.
ID
Requirement
Description
FR-6.1
Tie Detection
System automatically detects when the top two (or more) candidates for a position have equal vote counts after tallying
FR-6.2
Tie Flagging
Tied results are flagged as 'Unresolved ΓÇö Tie' on the results page instead of declaring a winner
FR-6.3
Runoff Trigger
Admin can trigger an automatic runoff election limited to the tied candidates, with a new short voting window
FR-6.4
Runoff Eligibility
Only voters from the original eligible roll for that election may vote in the runoff
FR-6.5
Audit Record
The original tie, the decision to run a runoff (or any alternative resolution), and the final outcome are all recorded in the immutable audit log
Hackathon demo note: a live runoff may not be practical to demo end-to-end in the time available. It is acceptable to demonstrate tie detection and flagging live, and explain the runoff workflow via the architecture/README rather than a full live re-vote.

5.5 Why This Matters for Judging
Production readiness: a real election system that crashes or guesses on a tie is not production-ready ΓÇö explicit, pre-configured rules close that gap
Security & integrity: locking the counting method and tie-break procedure before voting opens (and logging it) prevents any after-the-fact accusation of manipulating outcomes
Architecture: counting rules are configuration, not hardcoded logic ΓÇö the same Tally Service supports both Plurality and Majority elections without code changes, reinforcing the scalability story

8. Future Scalability Path
This MVP is intentionally scoped to COMPSSA, but every component is designed to generalize:
Multi-association support: the data model already separates Election from Organization, so any student body (not just COMPSSA) can run its own election on the same platform
University-wide elections: the same Voting and Tally services scale from hundreds to thousands of voters by adding API instances and read replicas, with no schema changes
Voter roll automation: voter upload can later integrate with HTU's official student records system instead of manual CSV upload
Verifiable receipts: the hash-chain design can be extended into a full cryptographic end-to-end verifiable voting scheme (e.g. mix-nets or homomorphic tallying) as a post-hackathon enhancement
